# STS2 BI Modelling

## 🎯 Overview

This project analyses performance in Slay the Spire 2 using run-level JSON data.

The goal is to understand:

* What drives win rate
* How runs evolve over time
* Which decisions impact outcomes

---

## 📂 Project Structure

```
sts2-run-analysis/
├── README.md
├── scripts/
├── data/
│   ├── raw/
│   ├── processed/
│   └── exploration/
```

---

## 📊 Data Source

Data comes from local `.run` files generated by the game.

Each file represents a single run and contains:

* Player data
* Map progression
* Events and outcomes

⚠️ Raw data is not included in this repository.

---

## 🧠 Approach

The project follows a structured workflow:

1. Explore JSON structure
2. Identify key entities and relationships
3. Build a simple analytical model
4. Expand iteratively

---

## 🧱 Data Model (Phase 1)

### `runs`

* One row per run
* Stores overall outcome and metadata

### `run_floor_state`

* One row per run per floor
* Tracks player state over time (HP, gold, etc.)

---

## 🧩 Key Concept

The data represents a **timeline of events and state**:

* Events = actions (combat, card choices, rewards)
* State = results after those actions (HP, gold)

---

## ⚙️ Scripts

Located in `/scripts/`:

* `extract_runs.py` → builds run-level dataset
* `extract_floor_state.py` → builds floor-level dataset

---

## 📁 Data Folders

* `data/raw/` → local game files (not tracked)
* `data/processed/` → generated datasets (not tracked)
* `data/exploration/` → structure analysis outputs

---

## 🚧 Current Scope

* Focused on single-player analysis
* Multiplayer is identified but not fully modelled
* Model is intentionally simple and will expand over time

---

## 🚀 Next Steps
